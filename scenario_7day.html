1 — Positioning

Spatial Memory is a room-scale AI memory manager for shared physical workspaces.

The room captures annotations in context, organises them via a Memory Graph, and surfaces the right note to each person at the right moment. The agent lives in the space, not on the body.

1

What it is — a room-scale AI memory manager

SM is not a tool you open. It is a persistent agent that lives in the workspace itself. Four properties define it:

  • Room-scale, not body-worn. The agent is embedded in the place, not on the person. People come and go; the room stays.
  • Multi-tenant. Many collaborators pass through the same space. The room remembers each separately and serves each differently.
  • Persistent. Memory accumulates across sessions, shifts, days, weeks.
  • Active, not passive. It is a manager: makes decisions about what to capture, how to organise, what to surface to whom — not a passive store.
2

What it inherits

Three intellectual lineages converge in SM:

  • Place over space (Harrison & Dourish 1996) — treats the workspace as socially meaningful place, not just coordinates. SM gives a structured way to encode that meaning via three orthogonal memory indexes.
  • Wearable AI agents (AiGet, SensibleAgent, ProMemAssist, VisionClaw, 2024–2025) — the proactive-AR-agent paradigm. SM relocates the agent from the person to the room. That displacement is itself a contribution.
  • Graph-organised knowledge (Obsidian, Roam, wiki lineage) — typed-link graphs of notes have a 30-year tradition. SM extends that to multi-user spatially-anchored async communication with proactive surfacing.
3

What it adds

Three technical contributions sit on top of the inherited lineages:

  • Memory Graph — a typed multigraph encoding three orthogonal indexes (Object / Project / Person). One annotation, three queries. Formal traversal operations with complexity bounds.
  • Selective-surfacing agent — decides what to play given who-you-are, where-you-are, what-you've-heard, and what-the-graph-knows. Multi-objective, ablatable, evaluatable.
  • Annotation modality TBD — could be voice memo, AR text sticky note, anchored gesture pin, or hybrid. Design decision still open. The Memory Graph and surfacing agent are modality-agnostic; whatever modality wins, the architecture absorbs it.
The system contribution sits at the intersection of all three: an AI manager that lives in the room, encoding social place via structured indexes, with annotations whose modality is an open design space.

3 — The concept: 3 interaction verbs × 3 memory types

SM's concept-level contribution is a 3×3 model. Three interaction verbs define how people act on the memory; three memory types define the indexes through which the memory is structured. Together they cover every interaction in the system. (Throughout this document, annotation is modality-agnostic — see §1, card 3 TBD.)

The three interaction verbs

Earlier framings (paper §3) called these "properties" — Anchored, Accumulated, Layered, Navigable. Re-read as actions, three verbs cover the same ground: Accumulated dissolves into Anchor performed by multiple people over time; Layered becomes Scope.

1

Anchor

Attach a memo to a place, object, person, or process. Multiple people anchoring over time = community memory accumulates at each place. This is the capture action.

absorbs: Anchored + Accumulated
2

Scope

Control visibility: audience (broadcast / directed / project), persistence (default / permanent / archivable), urgency (info / warning / blocking). Some info is always there for everyone; some is layered for specific eyes.

renamed from: Layered
3

Navigate

Retrieve memory by querying one of three orthogonal indexes — Object / Project / Person. The same Memory Graph; three lenses.

renamed from: Navigable

The three memory types (Navigate's three indexes)

The three memory types are what Navigate queries. The same Note can be indexed in all three.

🟧 Object memory

"What is this thing's history?"

Memory anchored to a workpiece, tool, machine, fixture, or material. Surfaces when you approach the object physically.

Examples: bed-leveling note on the 3D printer · drill-bit-dull tag · helping-grip ghost on the part · scrap-pile touch history

🟩 Project memory

"What is this work?"

Memory scoped to a Process — a multi-day, multi-person job. Surfaces when the user engages with the project, regardless of physical location.

Examples: WIP status · walking-tour recipe · departure retrospective · "parts arriving Wed"

🟪 Person memory

"What did they do?"

Memory indexed to a Person — their trail of notes, paths walked, techniques recorded. Surfaces on "follow" or when notes are tagged with you as recipient.

Examples: Alice's walking-tour route · Bob's station-default · Carol's Q-thread · "DIRECTED_TO Bob" handoff

A single annotation can be indexed by all three. "Carol, I left a fresh stencil at the consumables station — already loaded for tonight's run" is Anchored to obj (the stencil), tagged with prj (the wearable build), and Directed-To per (Carol).

Extending place-and-space (Harrison & Dourish 1996)

Harrison & Dourish distinguished place (socially meaningful) from space (physical coordinates). SM extends that distinction into a 3-axis indexing model: a Note has a place-coordinate (Object), a practice-coordinate (Project), and a person-coordinate (Person). The Memory Graph makes those three indexes queryable simultaneously.

How this compares to Obsidian / Roam

SM is structurally close to a wiki-graph knowledge tool like Obsidian or Roam, but extends to multi-user voice-based + spatially-anchored async communication with proactive surfacing.

5 — Why AR (three defense moves)

The 7-day flow above isn't defended by listing AR features Slack lacks. It's defended by three structural moves modeled on Fender & Holz's AsyncReality (CHI 2022) — but pushed further.

1

Re-frame async as the older form of shared-workspace communication

Place-anchored handoff predates Slack — workshop chalkboards, sticky notes on machines, whiteboard schedules, shop logs. These weren't worse than synchronous; they were more contextual because they lived where the work was. The rise of placeless digital channels (email, Slack) didn't improve async — it displaced async from the workspace. SM doesn't compete with synchronous tools; it restores an older mode with modern affordances.

2

Sidestep "just send a Slack message" — both sides

Slack and similar channels require explicit composition AND explicit retrieval. Both sides cost effort:

  • Sender: must verbosely describe the context the place would otherwise carry — "the part on the third shelf, back-left bolt, watch the hot end on the left side." Error-prone, time-consuming.
  • Recipient: must remember to open Slack, find the right channel, scroll to the relevant message, and translate text-about-place back to where-in-place.

SM removes both burdens. The voice memo is recorded in-place during work (no description needed — the place is the description). It surfaces proactively when the next collaborator arrives at that place (no remembering, no searching). AsyncReality avoided sender-side effort via auto-capture; SM extends to retrieval-side effort too. A two-sided sidestep, sharper than theirs.

3

Irreducible technical claim — Memory Graph + selective-surfacing agent

"Slack with location-aware push" would surface every message at every machine ever sent, overwhelming any user who walks in. SM's irreducible technical contribution is two things together:

  • The Memory Graph encodes semantic relationships a flat channel can't represent: dependency (warning before action), version (v2 supersedes v1), procedure (PROCEDURAL_NEXT for the walking-tour), thread, OPEN_QUESTION, CROSS_PROCESS_REF.
  • The selective-surfacing agent decides what to play given who you are (junior surfaces orientation; senior surfaces deltas), where you are, what you've already heard, and what the graph knows. AsyncReality's analog: causality-preserving playback.

A flat Slack channel can thread, but it cannot encode dependency / procedure / version semantically, cannot accept voice / gesture / gaze as semantic content with anchors, and cannot selectively surface. The graph + agent together are the technical contribution that makes accumulated spatial history navigable rather than overwhelming.

You don't browse, you don't remember, you don't recall — the place tells you.

That tagline is the SM move 2 in one line. The 7-day flow above demonstrates it across 37 primitives, with the Memory Graph + agent layer (move 3) keeping accumulated history tractable instead of noisy.

AsyncReality (CHI 2022) — single-user VR recovery

  • Sender effort: removed via auto-capture (RGB-D camera network)
  • Recipient effort: implicit — events replay when user returns
  • Irreducible: Causality Graph (event ordering)
  • Setting: single-user immersion-vs-real-world conflict

Spatial Memory — multi-user multi-day shared workspace

  • Sender effort: removed via in-place voice memo (hands-free during work)
  • Recipient effort: removed via proactive place-surfacing (no browse, no recall)
  • Irreducible: Memory Graph + selective-surfacing agent
  • Setting: multi-person, multi-day, async coordination

7 — Continuity Graph extensions forced by this scenario

The original SM v3 graph spec had four node types (Note / Location / Person / Object) and five edges (ANCHORED_TO / LEFT_BY / DIRECTED_TO / ABOUT / REPLIES_TO). The 7-day scenario forces the following extensions.

New node types

NodeWhat it representsWhy this scenario needs it
ProcedureOrdered sequence of Location + Note pairs (the walking-tour)Primitive #1 needs to be a graph object — replayable, versionable
SessionA maker's contiguous lab visit (start, end, notes left)"What did Carol contribute Wed?" needs Session as a query target
ProcessA multi-day, multi-person job tying notes / objects / proceduresThe wearable build is one Process; the neighbouring install team's job is another
Equipment (refines Location)Machine with state — current setup, last user, conditionMachine state handover, fixture drift, queue all need machine-as-stateful-node
Material (refines Object)Consumable with condition + remaining countLast-user condition log, material swap flag
Tool (refines Object)Movable item with last-borrower + return stateTool return note, scope-leads thread
IncidentSafety event with location, cause, resolution threadSafety incident echo (Carol's burn)

New edge types

EdgeWhat it representsWhere it shows up
PROCEDURAL_NEXTOrdering within a Procedure#1 walking-tour
UPDATESA note / Procedure supersedes a prior versionAlice's tour v2 (Day 5)
OPEN_QUESTIONA note seeking any answerer (no specific DIRECTED_TO)Reserved for broadcast queries with no recipient (not exercised by any retained canvas event)

Three formal contributions (vs. v3's one)

  1. Procedure as a first-class graph structure — sequence-with-anchors-and-versions, not just a flat note list. The "novice follows expert path" affordance is unrepresentable in a flat note graph.
  2. Process / Session scoping — temporal hierarchy (multi-day Process > Session > moment). Enables queries like "what happened during the wearable build" without knowing every individual note ID.
  3. Cross-Process linking — enables real shared-lab scenarios where projects occasionally borrow / coordinate (Day 6 scope-lead borrow).

This trebles the formal-contribution weight of §4 — was 22% of the paper, now justifies 25–28% with these additions.

8 — LLM-agent architecture forced by this scenario

The 7-day scenario can't be served by a single LLM doing everything. Agents decompose into four roles, each with its own slice of the graph.

Surfacer

When a maker arrives at a Location, decide what to render.

In: current Location, current Process, role of user (Carol = junior surfaces more orientation; Bob = mid surfaces less), recency, OPEN_QUESTION queue, DIRECTED_TO matches
Out: ranked list of voice memos to play

Threader

When a maker creates a voice memo, anchor it correctly.

In: spoken content (intent classification), current Location/Object, current Process, recently-played notes (likely reply target)
Out: graph edges — ANCHORED_TO, ABOUT, REPLIES_TO or new note, optional DIRECTED_TO

Tour Guide

Replay a Procedure with self-paced narration.

In: Procedure node, current user position, latest UPDATES versions
Out: spatial cues to guide user, voice memos at each anchor, override notifications ("Bob added a setup note since this was recorded")

Curator

Periodic summarization, conflict detection, stale-note flagging.

In: all notes within a Process or Session, time
Out: WIP status updates, "tour v1 superseded by v2" markers, deprecation flags

Plus a Routing sub-agent for OPEN_QUESTION threads — identifies likely answerers based on past notes at the same Location and notifies them passively (note surfaces when they next walk by).
